# QR Scanning — Performance Requirements & Target Corpus

Status: **requirements + benchmark hypothesis** (the two-mode split and the exact
resolution cutoff are to be confirmed by the Phase 1 sweep; the *targets* and
*formats* below are settled facts). Phase 1 lead repo: `esp-board-common`,
app `apps/qr_decoder`. See the cross-repo
`../seedsigner-micropython-builder/docs/camera-pipeline-integration-plan.md`.

This doc supersedes that plan's original "QR worst case" assumption
(2-of-3, v10–13, <70 modules, ~240² floor), which was derived before the real
Sparrow corpus was measured.

## 1. The real QR target corpus

QR side length in modules ⇒ version is `version = (modules − 17) / 4`. Test
fixtures generated by `btc-datagen` (Sparrow-fidelity); the two static ones are
checked into `apps/qr_decoder/test_qrs/`.

| Target | Modules | Version | Format | Tier |
|---|---|---|---|---|
| 2-of-3 multisig descriptor (static) | 77 | v15 | static QR | default (below baseline) |
| UR animated, small data | 81 | v16 | UR (fountain) | default (below baseline) |
| **UR animated, large/many-frame** | **85** | **v17** | UR (fountain) | **default — must-scan baseline** |
| BBQr, "low" cap | 89 | v18 | BBQr (fixed-part) | high-resolution mode |
| 3-of-5 multisig descriptor (static) | 93 | v19 | static QR | high-resolution mode |
| BBQr, max | 125 | v27 | BBQr (fixed-part) | high-resolution mode |

**Must-scan baseline = 85 modules (v17)** — the largest-data animated UR. The
default mode's design target is the *largest frame it must accept*, so 85
supersedes the static 2-of-3 (77) and small UR (81); both sit comfortably below
it.

## 2. Two operating modes (hypothesis)

- **Default / fast mode** — ceiling is **UR animated (≤85 modules)**. This is the
  everyday signing flow (PSBTs over UR). Optimise for fps + decode throughput.
  UR is fountain-coded, so dropped frames extend completion time, not fail it.
- **High-resolution mode (opt-in)** — denser, infrequent targets: BBQr (89–125)
  and static multisig descriptor backups (93). Larger square, deliberately lower
  fps; for animated BBQr the user lowers Sparrow's transmit rate (see §4).

The cut sits at ~85/89 modules. Whether two modes are actually warranted — and
exactly where the cutoff falls — is what the Phase 1 sweep decides (§6).

## 3. Decode-resolution requirements

- **Framing assumption:** the QR fills **~90%** of the capture square (the user's
  aiming target).
- **Resolution metric:** **≥3 pixels/module** is the rule-of-thumb decode floor.
  It is a *soft* floor — favorable lighting may resolve below it; the sweep
  measures the true empirical knee rather than assuming this number.
- `px/module = 0.90 × square / modules`. Floor square for N modules ≈ `3.33 × N`.

Decode floor for the **85-module baseline** (90% fill):

| Square (px) | px/module @ 85-mod | Reading |
|---|---|---|
| 240 | 2.54 | sub-floor |
| 280 | 2.96 | at the line |
| **320** | **3.39** | workable |
| 360 | 3.81 | comfortable |
| 480 | 5.08 | generous |

⇒ default-mode square lands around **320–360**; 240 is sub-floor, 280 marginal.
The static 77 / 93 fixtures bracket the 85 baseline (77 < 85 < 93), so its
resolution floor is interpolatable from them; the live UR animation tests it
directly at throughput.

**`K_QUIRC_MAX_VERSION` constraint:** the build is pinned to **25** — k_quirc's
version DB (`quirc_version_db`) is only populated through v25, and raising the
constant above that corrupts the heap on decode (zeroed version_info entries; see
`docs/knowledge/k-quirc-max-version-db-bound.md`). 25 (≤117 modules) clears
everything *except* **BBQr-max (v27)**. Supporting BBQr-max requires **first
extending the version DB** to v27+ (upstream quirc has the data through v40),
*then* raising the constant — a bump alone is not safe. Alternatively, constrain
users to BBQr "low" (89 / v18), which stays within 25.

## 4. Throughput requirements (animated QR)

For animated QR the binding constraint is **decode fps vs transmit fps** — NOT
preview-smoothness fps. (For a *static* QR, decodability is purely
resolution/lighting and is independent of the presentation layer.)

- **Sparrow transmit rate:** adjustable **0.5–10 fps, default 5**.
- **Decode-rate target:** **≥ ~10 decode fps** (≈2× the default 5) so capture
  isn't aliased against the unsynchronised display and each frame is caught.
  Measured quirc baseline was 8–13 decode fps at a 480 square; a ~320 default
  square is ~0.44× the pixels ⇒ decode fps should rise comfortably above 10.
  **This is the default-mode go/no-go: smallest square that reliably decodes 85
  modules *and* sustains ≥~10 decode fps on a live UR animation.**
- **Format resilience:** UR is fountain-coded (rateless) — dropped frames just
  delay reconstruction. BBQr is fixed-part — each unique part must be captured at
  least once, so it needs more decode-rate margin.
- **High-resolution mode + animation:** it deliberately tanks fps, so it pairs
  with the user **lowering Sparrow's transmit rate (≈0.5–2 fps)** for dense
  animated payloads.
- **Lighting/exposure:** freezing a frame that changes every 100 ms (10 fps)
  needs short exposure; dim light → long exposure → motion/transition blur. Good
  lighting helps *both* resolution and clean animated capture — the same lever.

## 5. Mode selection (UX, deferred to Phase 2+)

Mode is mostly a **programmatic** decision: most scan steps know their payload
type (a "scan your multisig descriptor" view → high-res; "scan PSBT" → default),
so the calling flow sets the mode — no user toggle. The lone ambiguous entry is
the **type-agnostic main-menu scan**; later options are a UI mode selector or
auto-escalation (start default, bump to high-res if nothing decodes within T s).
Not a Phase 1 concern — the spike only proves the two operating points exist.

## 6. Benchmark / measurement spec (Phase 1)

Run in `apps/qr_decoder` on the ESP32-P4 4.3" (DSI). Produce the **square ×
QR-density surface**:

- **Square knob:** `CONFIG_QR_DECODE_SQUARE` (0 = auto; sweep per build from
  sdkconfig, no source edits). Default-mode sweep focus: **{280, 320, 360}**
  (add 240/480 to bracket); high-res corner needs ≥480 + the MAX_VERSION bump.
- **Metrics (per build/target):** decode **reliability** and **throughput**:
  - `ok%` — % of frames that fully decoded → reliability headline.
  - `id%` — % of frames where the QR was located. The **id% vs ok% gap** is the
    diagnostic: high id% + low ok% = square locates the QR but can't resolve its
    modules (true decode knee); low id% = too small to even find it.
  - `decode fps`, `det/s`, plus pipeline `cam`/`disp` fps.
- **Instrumentation:**
  - Serial (every 2 s, `CONFIG_CAM_PIPELINE_QR_DEBUG`):
    `decode=…fps(gray=… quirc=…) det/s=… id=…% ok=…%`.
  - On-device **HUD** (DSI): `sq<N>  cam… disp… dec…` / `id…% ok…%  det…/s`,
    plus a compact per-decode flash (`✓ #<count>  <bytes> B`) to eyeball the
    decode rate against the animation.
- **Reliability protocol:** decode reliability is a *rate*, not a binary — fix a
  protocol per point (e.g. hold the fixture for a fixed window under fixed
  lighting; read steady-state `ok%`).
- **Read-offs:** the **cutoff** = largest module-count a small (≈320) square
  decodes reliably at ≥10 fps; the **marked-difference** test = fps gap between
  the default square and the high-res square (decides whether two modes pay for
  themselves).
- **Isolating resolution from presentation:** static decodability is
  presentation-independent, so the true decode floor can be measured with the
  preview minimized (removes PSRAM contention); fps cost is then measured
  separately with the preview on.

### Event-based metrics (preferred over per-frame %)

`ok%`/`id%` have a polluted denominator (all the frames where the camera isn't
aimed at a QR), so they understate and jitter. The clean signal is the stream of
**successful-read events** and the **time between them** — mined from the
timestamped `QR decoded` / `QRPX` serial lines, no frame-counting needed:
- **Inter-read interval (ms)** between consecutive decodes → effective decode
  rate and its consistency.
- **Animated distinct-parts/sec** (each UR part is unique) → the throughput that
  must keep up with Sparrow's 5 fps. *This is the headline animated metric.*

### Measured px/module + distance sweep (the real decode floor)

Each decode logs its **actual** resolution, computed from quirc's reported corner
pixel coordinates (`k_quirc_result_t.corners`) and version: `QRPX … N.NNpx/mod`.
This replaces the assumed-90%-fill estimate with ground truth. Procedure: at a
**fixed square**, slowly move the device closer/farther while scanning — this
sweeps the QR's pixel footprint optically (quirc cost constant), producing a
continuous **decode-success vs measured-px/module** curve that reveals the
empirical decode floor and the focus envelope. Complements the square sweep
(which varies throughput); together they give the full reliability × throughput
surface. Static QRs are cleanest (dwell per distance); animated works if moved
slowly.

## 7. Performance requirements — summary

| Requirement | Target |
|---|---|
| Default-mode resolution | reliably decode **85-module / v17** frames (≈320–360 square) |
| Default-mode throughput | **≥ ~10 decode fps** to sustain UR animated at Sparrow default 5 fps |
| Default-mode coverage | UR animated (≤85), static 2-of-3 (77) incidentally |
| High-res-mode resolution | decode up to **125-module / v27** BBQr (requires `K_QUIRC_MAX_VERSION ≥ 27`) |
| High-res-mode throughput | low fps accepted; animated BBQr pairs with slowed transmit (0.5–2 fps) |
| Framing / lighting | ~90% fill; ≥3 px/module soft floor; good lighting aids both resolution and animated capture |

## 8. Future research areas (not in Phase 1 scope)

- **Perspective skew (priority).** Off-axis capture foreshortens the QR so
  px/module varies across the code and the grid sampling degrades. More likely in
  real handheld use than rotation, and a plausible decode-floor limiter — worth
  characterizing (decode reliability vs tilt angle) once the head-on floor is
  settled.
- **In-plane rotation (lower priority).** Users can be expected to aim roughly
  parallel, and quirc is rotation-tolerant by design, so this is a lower-value
  check — note for completeness, test after skew.
- **Sensor-mode sweep.** RAW8 800×640 vs the current 1280×960-binning, once the
  `VIDIOC_S_FMT` engine hook exists (see the integration plan).
- **High-density mode + BBQr-max.** Requires extending the k_quirc version DB
  past v25 (see `docs/knowledge/k-quirc-max-version-db-bound.md`).
- **Manual focus slider.** The OV5647 module supports a VCM focus motor
  (`CONFIG_CAMERA_OV5647_ISP_AF_ENABLE`) and the V4L2 stack exposes
  `V4L2_CID_FOCUS_ABSOLUTE`; our CSI driver currently leaves `set_focus = NULL`.
  Deliberately NOT doing autofocus (problematic). Future: wire a manual
  `set_focus` + a UI slider for fixed-distance focus tuning. (On-hardware: confirm
  the specific module has the VCM populated.)

## 9. Exposure (implemented)

Auto-exposure runs (ISP pipeline controller + IPA), but the AE *setpoint* was the
sensor/ISP default (`ae_target = 0`), which over-exposes bright backlit screens —
QR module highlights clip and decode fails at close range even with ample
px/module (a likely cause of the 3-of-5 inverted-distance oddity). Fixed via
`CONFIG_BOARD_CSI_AE_TARGET` (default 70 for `qr_decoder`); tune per build. The
RAW10 1280×960 binning mode also adds AE oscillation / harsh contrast — revisit
with the sensor-mode sweep.
